
# BackupDiveComputer Log Format Policy

> Status: Draft / v1.3.7-dev  
> Scope: Internal dive log storage, DiveSample/DiveEvent persistence, BLE log transfer policy, and future Subsurface-compatible export policy.  
> Safety Notice: This project is an experimental prototype and must not be used for real diving.

---

## 1. Purpose

This document defines the logging policy for BackupDiveComputer.

The purpose of the log system is to preserve the original dive data generated by the firmware in a compact, robust, versioned native format. This native log is the source of truth for later review, debugging, BLE transfer, and export to external dive log applications such as Subsurface.

The project shall avoid ambiguity between:

1. The internal firmware log format.
2. BLE transfer format.
3. Future Subsurface-compatible XML/UDDF/CSV export format.
4. UI display priority during a dive.

---

## 2. Core Policy

### 2.1 Native binary log is the source of truth

The dive computer shall store dive logs internally using the project’s own native binary format.

The native log format is defined by:

- `DiveLogHeader`
- `DiveSample`
- `DiveEvent`

The native log file is the authoritative record of a dive.

Future XML, UDDF, or CSV files are considered exported representations generated from the native log. They are not the primary storage format.

---

### 2.2 XML shall not be the primary on-device storage format

The firmware shall not use Subsurface XML, UDDF, or CSV as the primary on-device storage format.

Reasons:

- XML is larger than the native binary format.
- XML generation is memory-intensive on ESP32-class devices.
- XML is more vulnerable to corruption if power is lost during write.
- XML escaping and string handling increase firmware complexity.
- Subsurface XML schema compatibility may change over time.
- Native logs allow later re-export to multiple formats.

Therefore:

```text
Internal storage: BDC native binary log
BLE transfer:     raw BDC native log first
External export:  BDC native log -> Subsurface XML / UDDF / CSV
```

---

## 3. Recommended Data Flow

The preferred architecture is:

```text
ESP32-S3 firmware
 ├─ Records dive data
 ├─ Stores native .bdc log in LittleFS or RAM fallback
 └─ Exposes log through BLE as raw native data

External tool / mobile app / desktop utility
 ├─ Downloads .bdc log over BLE
 ├─ Parses native BDC log
 ├─ Converts to Subsurface XML, UDDF, or CSV
 └─ User imports exported file into Subsurface
```

The firmware should prioritize reliable recording over direct XML generation.

---

## 4. File Structure

The native dive log file should be structured as:

```text
[ DiveLogHeader ]
[ DiveSample sampleCount entries ]
[ DiveEvent  eventCount entries ]
```

The exact C/C++ definitions are maintained in:

```text
include/log_format.h
```

The documentation shall describe the policy and compatibility requirements, while `log_format.h` remains the source for exact field layout.

---

## 5. DiveLogHeader Policy

`DiveLogHeader` stores dive-level metadata.

Expected fields include, but are not limited to:

- magic
- version
- diveNumber
- timeStatus
- bootCount
- startEpochSec
- endEpochSec
- durationSec
- noFlyEndEpochSec
- gasFo2Percent
- ppO2MaxCentiBar
- decoViolation
- postViolationAdvisory
- reentryCount
- missedStopDepthCm
- missedStopRemainSec
- advisoryEndEpochSec
- maxDepthCm
- avgDepthCm
- minTempDeciC
- sampleCount
- eventCount
- gpsLatE7
- gpsLonE7
- gpsValid

The header must always include:

```text
magic
version
sampleCount
eventCount
```

These fields are required for compatibility and recovery.

---

## 6. DiveSample Policy

`DiveSample` stores time-series dive profile data.

Minimum expected sample fields:

```cpp
struct DiveSample {
    uint32_t timeSec;
    int16_t depthCm;
    int16_t tempDeciC;
    int16_t ndlOrTtsMin;
};
```

### 6.1 Sampling policy

Dive samples should be recorded at the configured log interval.

Current expected interval:

```text
LOG_INTERVAL_MS = 4000 ms
```

The sample interval may change in future firmware versions, so import/export tools must not assume a fixed interval. They should use `timeSec`.

### 6.2 Depth units

Depth shall be stored in centimeters.

```text
depthCm = depth in meters * 100
```

Examples:

```text
3.0 m  -> 300 cm
18.5 m -> 1850 cm
```

### 6.3 Temperature units

Temperature shall be stored in deci-degrees Celsius.

```text
tempDeciC = temperature in Celsius * 10
```

Examples:

```text
24.5 C -> 245
18.0 C -> 180
```

### 6.4 NDL/TTS field

`ndlOrTtsMin` may represent:

- NDL minutes during normal no-decompression phase.
- TTS minutes during decompression phase.

The interpretation depends on dive state and event context.

Future versions may add explicit fields for:

- ceiling depth
- current deco stop depth
- remaining deco stop seconds
- ascent rate
- ppO2
- GF99

If such fields are added, the log version must be incremented.

---

## 7. DiveEvent Policy

`DiveEvent` records important state transitions, warnings, violations, and operational events.

Minimum expected structure:

```cpp
struct DiveEvent {
    uint32_t timeSec;
    uint8_t type;
    int32_t value;
};
```

### 7.1 Event time

`timeSec` is relative to the dive start.

Example:

```text
timeSec = 420
```

means the event occurred 7 minutes after dive start.

### 7.2 Event value

`value` is event-specific.

Examples:

```text
DECO_REQUIRED:       stop depth in cm or stop depth in m depending on implementation
DECO_STOP_CHANGED:   new stop depth
CEILING_EXCEEDED:    current ceiling or missed stop depth
ASCENT_RATE_WARN:    ascent rate in cm/min or scaled m/min
LOW_BATTERY:         battery percentage
```

The exact value encoding for each event type must be documented whenever new event types are added.

---

## 8. Required Event Types

The firmware should persist the following event classes.

Exact numeric constants are defined in:

```text
include/log_format.h
```

### 8.1 Decompression events

Required:

```text
EVENT_DECO_REQUIRED
EVENT_DECO_CLEARED
EVENT_DECO_STOP_CHANGED
EVENT_CEILING_EXCEEDED
EVENT_DECO_VIOLATION
```

These events are required because decompression behavior is safety-critical and must be reviewable after the dive.

### 8.2 Safety stop events

Required:

```text
EVENT_SAFETY_STOP_START
EVENT_SAFETY_STOP_DONE
EVENT_SAFETY_STOP_SKIPPED
```

Safety stop events shall be recorded separately from decompression stop events.

However, Safety Stop must never override DECO.STOP.

### 8.3 Ascent and warning events

Recommended:

```text
EVENT_ASCENT_RATE_WARN
EVENT_LOW_BATTERY
EVENT_PPO2_HIGH
EVENT_MOD_EXCEEDED
```

These events are useful for post-dive review and future XML export.

## 9. DECO.STOP and Safety Stop Logging Policy

### 9.1 DECO.STOP priority

DECO.STOP always has priority over Safety Stop.

If a decompression obligation exists, the firmware must remain in DECO mode and must not switch to Safety Stop mode.

This applies even when the diver is within the usual Safety Stop depth range.

Correct behavior:

```text
If DECO is required:
    show DECO.STOP
    run DECO timer when within stop window
    do not enter S.STOP
```

Incorrect behavior:

```text
If diver reaches 3m-6m:
    switch from DECO.STOP to S.STOP while deco remains
```

This is prohibited.

---

### 9.2 Safety Stop after DECO

After DECO.STOP has been completed, the firmware shall not force a separate Safety Stop as an additional mandatory stop.

Rationale:

- The diver has already completed a decompression stop.
- The final decompression stop functionally satisfies or supersedes the recreational Safety Stop requirement.
- The UI should avoid suggesting that a new separate S.STOP is required after DECO clears.

Therefore:

```text
DECO completed -> no forced separate S.STOP
```

If a future optional policy allows a post-deco voluntary safety stop, it must be explicitly documented and must not be presented as mandatory.

---

### 9.3 DECO stop display actions

For a 3m final stop with the configured margins:

```text
target stop depth: 3.0 m
shallow margin:    0.6 m
deep margin:       1.8 m
```

Expected display:

```text
depth < 2.4 m       -> DESCEND
2.4 m to 4.8 m      -> HOLD
depth > 4.8 m       -> ASCEND
```

At 3.7m during a 3m DECO.STOP, the UI shall display:

```text
HOLD
```

not:

```text
ASCEND
```

because the diver is within the valid stop window and the timer should continue running.

## 10. DECO Timer Policy

### 10.1 Second-based timer

DECO stop time shall be tracked in seconds.

The firmware must not rely only on minute-based `stop_time_min` for the active stop countdown.

Required field:

```cpp
stop_time_sec
```

`stop_time_min` may still be kept as a rounded display/helper field, but the actual countdown must use seconds.

---

### 10.2 20-second rounding policy

Displayed and enforced DECO stop time shall be rounded up to the next 20-second unit.

This is intentionally conservative.

Examples:

```text
actual required stop time: 1:00 -> displayed/enforced: 1:00
actual required stop time: 1:01 -> displayed/enforced: 1:20
actual required stop time: 1:02 -> displayed/enforced: 1:20
actual required stop time: 1:20 -> displayed/enforced: 1:20
actual required stop time: 1:21 -> displayed/enforced: 1:40
```

The rounding must always round upward, not to the nearest 20 seconds.

Formula:

```cpp
roundedSec = ((requiredSec + 20 - 1) / 20) * 20;
```

---

### 10.3 Timer reset prevention

The DECO timer must not reset from `0:00` back to `1:00` or any other value unless a new decompression obligation is calculated and explicitly entered.

Prohibited behavior:

```text
DECO timer reaches 0:00
firmware sees stop_time_min > 0
firmware resets countdown to 1:00
```

Required behavior:

```text
DECO timer reaches 0:00
ceiling is recalculated
if obligation cleared:
    DECO clears
else:
    next required stop time is calculated in seconds and rounded up
```

### 10.4 Full stop-duration prediction

The DECO stop timer shall represent the predicted total time required at the current stop before the model allows transition to the next shallower stop or DECO clear.

The firmware must not repeatedly display short 20-second fragments when a longer stop is required.

Prohibited behavior:

```text
DECO.STOP 12m 0:20
countdown reaches 0:00
model still requires 12m
timer resets to 0:20
repeats multiple times
```

Required behavior:

```text
If the current stop requires approximately 60 seconds:
    display 1:00 from the beginning

If the current stop requires approximately 80 seconds:
    display 1:20 from the beginning
```

The Bühlmann calculation may simulate future off-gassing using a copy of the current tissue state. The real tissue state must not be modified by timer prediction.

## 추가 섹션: conservative depth

```md
### 10.5 Conservative stop calculation depth

The displayed DECO stop depth remains the nominal ladder depth.

Example:

```text
Displayed stop: DECO.STOP 12m
```

However, timer prediction uses the deepest valid HOLD depth:

```text
effectiveCalcDepthM = stopDepthM + DECO_STOP_DEEP_MARGIN_M
```

Examples:

```text
18m stop -> calculate at 19.8m
15m stop -> calculate at 16.8m
12m stop -> calculate at 13.8m
9m stop  -> calculate at 10.8m
6m stop  -> calculate at 7.8m
3m stop  -> calculate at 4.8m
```

Rationale:

- Avoid timer jitter from small depth changes caused by current, buoyancy, or sensor noise.
- Preserve conservative behavior for a backup dive computer.
- Keep the displayed stop depth simple and consistent with the DECO ladder.


---

### 10.6 Displayed rounded timer is authoritative

Once a rounded DECO stop timer is displayed, the diver must complete the displayed countdown before moving to the next shallower stop or clearing DECO.

Example:

```text
Model-required time: 1:01
Rounded display:     1:20
Required behavior:   stay for 1:20
```

The firmware must not transition to the next stop at 1:01 simply because the live model now allows the transition.

Required transition condition:

```text
model allows shallower stop or clear
AND
displayed local countdown has reached 0
```

This preserves the conservative 20-second round-up policy.


---

### 10.7 Resync policy

A DECO timer resync is allowed only as a diagnostic safety net.

Normal behavior should not require repeated resyncs.

Acceptable:

```text
One occasional resync due to calculation/timing mismatch
```

Suspicious / should be investigated:

```text
Repeated resync at the same stop depth
Repeated 0:20 reset at the same stop depth
Large model remain value restored after local countdown reached 0
```

The `Resync` Serial log is intentionally kept visible because it indicates a condition that should be reviewed during simulation.

## 11. Final Stop Depth Policy

The final decompression stop depth shall be configurable at compile time.

Recommended configuration macro:

```cpp
#define DECO_LAST_STOP_DEPTH_M 3
```

Allowed values:

```text
3
6
```

Invalid values should produce a compile-time error.

Expected behavior:

### 3m final stop

```text
18m -> 15m -> 12m -> 9m -> 6m -> 3m -> surface
```

### 6m final stop

```text
18m -> 15m -> 12m -> 9m -> 6m -> surface
```

When `DECO_LAST_STOP_DEPTH_M` is set to 6, the 3m decompression stop is skipped.

## 12. BLE Transfer Policy

### 12.1 First priority: raw native log download

The first BLE log transfer implementation should expose the raw native BDC log.

Recommended BLE characteristics:

```text
Log List
Log Metadata
Log Data Chunk
Log Transfer Control
```

The external client should reconstruct the `.bdc` file from chunks.

### 12.2 XML over BLE is not the first priority

The firmware may support XML generation in the future, but it is not the preferred first implementation.

Reasons:

- Larger transfer size.
- More firmware complexity.
- XML schema risk.
- More memory usage.
- Harder to debug than raw native logs.

Recommended future approach:

```text
BLE downloads .bdc
External app converts .bdc to XML/UDDF/CSV
```

---

## 13. Subsurface Export Policy

### 13.1 Export is derived from native log

Subsurface-compatible files shall be generated from native BDC logs.

The native log remains the source of truth.

Recommended export order:

```text
1. Subsurface native XML
2. UDDF
3. CSV profile fallback
```

Subsurface documentation states that Subsurface supports multiple import paths including dive computer import, CSV import, UDDF, and other log formats.

References:

- https://subsurface-divelog.org/subsurface-user-manual/
- https://subsurface-divelog.org/faq/

---

### 13.2 Initial Subsurface XML mapping

A minimal Subsurface XML export may map BDC data as follows:

```text
DiveLogHeader.diveNumber      -> dive number
DiveLogHeader.startEpochSec   -> dive date/time
DiveLogHeader.durationSec     -> dive duration
DiveLogHeader.maxDepthCm      -> maximum depth
DiveLogHeader.avgDepthCm      -> mean depth
DiveLogHeader.minTempDeciC    -> water temperature
DiveLogHeader.gasFo2Percent   -> gas O2 percentage
DiveLogHeader.gpsLatE7        -> dive site latitude
DiveLogHeader.gpsLonE7        -> dive site longitude
DiveSample[]                  -> profile samples
DiveEvent[]                   -> notes or profile events
```

A minimal XML profile may contain:

```xml
<divelog program='subsurface' version='3'>
  <dives>
    <dive number='1' date='YYYY-MM-DD' time='HH:MM:SS' duration='NN:SS min'>
      <divecomputer model='BackupDiveComputer'>
        <depth max='18.0 m' mean='10.5 m' />
        <temperature water='24.5 C' />
        <sample time='0:00 min' depth='0.0 m' temp='24.5 C' />
        <sample time='0:04 min' depth='1.2 m' temp='24.5 C' />
      </divecomputer>
    </dive>
  </dives>
</divelog>
```

---

### 13.3 Event export strategy

Initially, BDC events may be exported into XML notes for maximum compatibility.

Example:

```xml
<notes>
BackupDiveComputer events:
00:12:40 DECO REQUIRED stop=3m remain=1:20
00:13:20 CEILING EXCEEDED depth=2.1m
00:14:00 DECO CLEARED
</notes>
```

Later, once Subsurface profile event compatibility is verified, the exporter may emit proper profile event tags.

The native BDC event log must preserve enough information to support both approaches.

## 14. UI Priority Policy

The UI priority policy is related to logging because displayed warnings and state transitions should also be reflected in DiveEvent records.

Recommended in-dive UI priority:

```text
1. Critical hardware/system warning
2. PPO2 HIGH / MOD EXCEEDED
3. CEILING EXCEEDED
4. DECO.STOP
5. MISSED DECO / DECO.VIOL advisory
6. Safety Stop
7. Ascent-rate warning
8. NDL / normal dive information
9. Surface / PostDive / Charging
```

Important rule:

```text
DECO.STOP must always be displayed before Safety Stop.
```

If both DECO.STOP and Safety Stop conditions appear to be true, DECO.STOP wins.

---

## 15. Versioning Policy

The native log format must be versioned.

If a change modifies binary layout, field size, field order, or interpretation, the log version must be incremented.

Examples requiring version increment:

```text
Adding fields to DiveLogHeader
Changing DiveSample layout
Changing DiveEvent layout
Changing unit encoding
Changing event value encoding
```

Examples not necessarily requiring version increment:

```text
Adding new event type using existing DiveEvent layout
Changing UI text
Changing XML exporter formatting
Changing BLE chunk size
```

Import/export tools must check:

```text
magic
version
```

before parsing a log.

---

## 16. Recovery and Compatibility

The firmware and external tools should reject or safely ignore logs with:

```text
invalid magic
unsupported version
impossible sampleCount
impossible eventCount
truncated file
```

If a log is partially corrupted, recovery tools may attempt to parse valid samples/events up to the last complete record, but firmware should not rely on corrupted logs for operational state.

## 18. Acceptance Criteria

The logging system is considered acceptable when:

```text
1. A completed dive stores header + samples + events.
2. DECO entry and clear events are persisted.
3. Safety Stop start/done/skipped events are persisted.
4. Ceiling violation events are persisted.
5. Reboot after dive can reload and print the last dive log.
6. Raw native log can be transferred or extracted.
7. Native log contains enough data to generate Subsurface-compatible XML later.
8. UI priority is consistent with event records.
```

---

## 19. Non-goals for Current Firmware Stage

The following are not required in the first implementation:

```text
Direct on-device Subsurface XML generation
Direct on-device UDDF generation
Direct upload into Subsurface over BLE using libdivecomputer protocol
Cloud synchronization
Multi-dive logbook management beyond current storage scope
```

These may be considered in future versions after native log persistence is stable.

---

## 20. Summary

The BackupDiveComputer logging policy is:

```text
Native binary BDC log first.
XML/UDDF/CSV export later.
BLE transfers raw native logs first.
DiveEvent persistence is required before reliable export.
DECO.STOP always overrides Safety Stop.
DECO completion must not force a separate Safety Stop.
UI warnings and state transitions should be reflected in persisted events.
```

This policy minimizes firmware complexity, improves reliability, and preserves enough information for future Subsurface-compatible export.
